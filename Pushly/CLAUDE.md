# CLAUDE.md

このファイルは、Pushly プロジェクトで Claude Code および Codex 等の AI コーディングアシスタントが作業する際に従うべきルール・前提・禁止事項を定めるものです。

新規にコードを生成・変更する前に、必ずこのファイルの内容を確認してください。

---

## 1. プロジェクト概要

- **アプリ名**: Pushly
- **種別**: iOS 専用ネイティブアプリ
- **技術スタック**: Swift / SwiftUI / SwiftData / iOS 17+
- **同期**: CloudKit (private database)
- **永続化**: 端末ローカル + CloudKit のみ。外部サーバー・外部 API は使わない
- **用途**: 免許更新、サブスク更新、健康診断など、忘れがちな予定の通知
- **要件定義**: `Notify_要件定義書_v0.5.docx`
- **実装手順**: `prompt.md`

詳しい仕様の判断に迷ったら、まず上記2つのドキュメントを優先する。

## 2. 重要：秘密情報の取り扱い

### 絶対に書き込まない / コミットしない

以下の情報を**ソースコードに直接書かない**。万一見つけても削除する：

- API キー、シークレットキー、トークン
- パスワード、SSH 鍵、証明書ファイル
- Apple ID、メールアドレス、電話番号などの個人情報
- iCloud Container ID 以外の Apple Developer アカウント情報
- バンドル ID は Xcode の設定ファイル(`.xcodeproj`)経由で管理し、コード中にハードコードしない
- ユーザーの個人データやテスト用に作成した実在の個人情報

これらが必要な場合は：

- **環境変数や Info.plist の設定キー経由で読み込む**
- **`.gitignore` に追加してリポジトリに含めない**

### 既存ファイルに秘密が混入していたら

- ユーザーに即座に報告する
- そのファイルを git からの追跡から外す方法を提案する
- 漏洩した可能性を伝え、キーのローテーションを推奨する

## 3. 通信・データ取り扱いポリシー

このアプリは「端末完結 + iCloud 同期のみ」が大原則です。以下は**絶対に行わない**：

- 外部 API への通信処理を追加する（HTTP/HTTPS リクエスト含む）
- アナリティクス SDK（Firebase、Mixpanel、Adjust 等）の導入
- 広告 SDK の導入
- 第三者にデータを送信するライブラリの追加
- ユーザーの位置情報、連絡先、カレンダー、写真等への不要なアクセス
- WebView の埋め込み（外部コンテンツ表示用）

ユーザーが指示しても、上記を実装する場合は**理由を確認してから**にしてください。仕様外の通信処理を勝手に追加しないこと。

### URL を扱う場合

ユーザーが入力した URL（Event の `urlString` フィールド）は：

- 表示のみ（タップしたら標準ブラウザで開く）
- アプリ内で fetch せず、内容も解析しない
- スキームチェック（http/https のみ許可）を必ず行う

## 4. 外部ライブラリ・パッケージの追加

### 原則：最小限に保つ

MVP（v1.0）では**追加ライブラリ不要**。Apple 標準フレームワークだけで実装できる：

- SwiftUI（UI）
- SwiftData（永続化）
- UserNotifications（通知）
- Swift Charts（グラフ、iOS 16+）
- CloudKit（同期、自動で組み込まれる）

### ライブラリを追加したい場合

新しい Swift Package を追加するときは、**事前にユーザーに確認**してください。確認時に伝える項目：

- パッケージ名と用途
- メンテナンス状況（最終更新、Star 数、Issue 数）
- ライセンス（MIT / Apache 等の許容可能なもの）
- 代替案（自作 / 標準API で済む可能性）

許容しないライブラリの例：

- ライセンスが不明 or 商用利用不可
- 1年以上メンテされていない
- 依存関係が極端に多い
- 用途と無関係な権限を要求する

## 5. コード生成時のルール

### Swift / SwiftUI のスタイル

- **Swift 5.9+ の機能**を使う（async/await、Macros 含む）
- **強制アンラップ（`!`）は避ける**。やむを得ない場合はコメントで理由を書く
- **`fatalError` は MVP 段階の限定的なエラーハンドリングに限る**。本番ではユーザーに見せられるエラーに置き換える
- **マジックナンバーを避け、定数化する**
- **ファイル冒頭に著作権コメント等は不要**

### SwiftData

CloudKit 同期との互換性のため、以下を厳守：

- 全プロパティに**デフォルト値**を設定
- `@Attribute(.unique)` は使わない
- リレーションは必ず **optional 型**
- リレーションには必ず **inverse を指定**

これに違反する変更を加えると、CloudKit 同期が壊れます。Phase 1 のデータモデル仕様（`prompt.md` 参照）を必ず守ること。

### 命名規則

- 型名: PascalCase（`Template`, `NotificationRule`）
- 変数・関数: camelCase（`scheduledDate`, `daysFromNow`）
- enum case: lowerCamelCase（`.scheduled`, `.completed`）
- 日本語のコメントは OK、ただし型名・変数名は英語で

### ローカリゼーション

- **MVP は日本語のみ対応**。多言語化は v1.0 では考えない
- 表示文字列は直接 Swift コード内に書いて OK
- `Localizable.strings` の整備は将来検討

## 6. ファイル操作の安全性

### 既存ファイルを変更する前に

- 必ず内容を `view` で確認してから編集する
- 同名のファイルを上書きする前に、既存版の役割を確認する
- 大量の削除や全置換は事前にユーザーに合意を取る

### 削除してはいけないもの

- `.xcodeproj` ディレクトリの内部ファイル
- Xcode 自動生成の `Info.plist`、`Assets.xcassets`
- `Preview Content/` ディレクトリ
- 既存のユーザー作成ファイル（Models/, Views/ 配下）

### 削除する場合

`git rm` を使い、コミットメッセージで明示する。物理削除前にユーザーに確認。

## 7. テストとビルド

### ビルド確認

- コード変更後は **必ずビルドが通ることを確認する**
- ビルドエラーをユーザーに渡す前に、自分で読んで原因を提示する
- ⌘B でビルドが通らない状態で「完了しました」と言わない

### Xcode プロジェクトファイルの編集

- `Pushly.xcodeproj/project.pbxproj` は手動編集しない
- ファイル追加は Xcode 経由か、PBXProject ライブラリ経由で安全に行う
- 構造を破壊しないよう、追加・削除前に diff を確認する

---

## 8. データの扱いとプライバシー

### ユーザーデータ

このアプリは個人の予定・金額・URL を扱います。以下を遵守：

- データはユーザーの **端末と iCloud 個人領域のみ** に保存
- ログ出力（`print`, `os_log`）に個人情報を含めない
- デバッグ用のサンプルデータに**実在の人名・電話番号・メール**を使わない
- 金額や URL を含むデータを画面外（コンソール、外部）に流さない

### デバッグログ

開発中の `print` 文は許可するが、リリースビルドでは無効化する仕組みを入れる：

```swift
#if DEBUG
print("Schedule notification: \(event.name)")
#endif
```

機密性のあるフィールド（金額、URL、メモ）は `#if DEBUG` 内でも出力を避ける。

## 9. リリース前のチェック

Phase 9 直前に以下を確認してから出荷：

- [ ] ハードコードされた個人情報、シークレットがない
- [ ] `print` や `os_log` の本番出力が抑制されている
- [ ] 不要な NSAppTransportSecurity 設定がない（外部通信不要なので）
- [ ] Info.plist の権限説明文（NSUserNotificationsUsageDescription 等）が日本語で適切
- [ ] App Sandbox / 必要最小限の Entitlements のみ有効
- [ ] `.gitignore` に `.DS_Store`, `xcuserdata/`, `*.xcscheme` 等が入っている
- [ ] git ログに秘密情報が混入していない

## 10. ユーザーとのコミュニケーション

### 仕様の解釈に迷ったら

- まず `Notify_要件定義書_v0.5.docx` を確認
- 該当しなければ `prompt.md` の対応 Phase を確認
- それでも不明なら**ユーザーに質問してから実装する**
- 推測で大規模な変更を進めない

### 危険な操作の前に

以下の操作は**ユーザーの明示的な許可**を取ってから実行：

- 既存ファイルの大量削除
- `git reset`, `git push --force`, ブランチ削除
- データベースのマイグレーション（SwiftData の Schema 変更）
- Capabilities の追加・削除
- バンドル ID の変更

### 報告のフォーマット

実装完了時の報告には以下を含める：

1. **何を変更したか**（ファイル名と概要）
2. **なぜそうしたか**（仕様根拠）
3. **テストの方法**（手動確認のステップ）
4. **想定される副作用**（ある場合）

## 11. 禁止事項まとめ

絶対に行わないこと：

- ❌ API キー・シークレットをコードに直書き
- ❌ 外部 API への通信処理を追加
- ❌ アナリティクス・広告 SDK の導入
- ❌ ユーザー個人情報のログ出力
- ❌ `.xcodeproj` の手動破壊的編集
- ❌ ユーザーの許可なく大量ファイル削除
- ❌ CloudKit 制約に違反する SwiftData モデル変更
- ❌ 推測ベースの大規模リファクタリング
- ❌ ビルドが通らない状態での「完了」報告

- セキュリティ上の問題を発見した → ただちに作業を止めてユーザーに報告
- 仕様と要件定義に矛盾を発見 → 質問してから判断
- 自分の出力に自信がない → 「これでいいか確認したい」と聞く
